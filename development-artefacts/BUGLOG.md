## BUG-001 — Анонимный доступ к каталогу проектов

**Дата обнаружения:** 2026-04-19
**Статус:** ✅ Исправлен
**Обнаружил:** Ручное тестирование в браузере

### Проявление
Пользователь, не прошедший авторизацию, открывал страницу `/projects/` и видел полный каталог проектов платформы без каких-либо ограничений.

### Корневая причина
View `project_list` и `project_detail` не имели декоратора `@login_required`. Платформа задумана как закрытая (для зарегистрированных участников), однако технического ограничения доступа не было.

### Исправление
Файл: `src/web/apps/frontend/views/projects.py`
- На `project_list` добавлен `@login_required(login_url="/auth/")`
- На `project_detail` добавлен `@login_required(login_url="/auth/")`

Django автоматически добавляет `?next=<текущий URL>` к редиректу, поэтому после входа пользователь возвращается туда, куда шёл.

### Нужно ли повторное внимание
Нет. Все публичные endpoint'ы кроме `/auth/` теперь закрыты. Если в будущем понадобится частично-публичный доступ (например, landing-страница с описанием платформы) — добавить отдельный view без `@login_required`.

## BUG-002 — HTML страницы входа вставлялся внутрь карточки проекта

**Дата обнаружения:** 2026-04-19
**Статус:** ✅ Исправлен
**Обнаружил:** Ручное тестирование в браузере

### Проявление
Неавторизованный пользователь открывал модальное окно «Откликнуться», вводил произвольный текст и нажимал «Отправить». После нажатия внутри карточки проекта появлялась форма логина (поля «Логин» и «Пароль») вместо нормальной реакции системы.

### Корневая причина — цепочка событий
1. Shared apply modal отправляет данные через `fetch()` (не HTMX) на `POST /projects/<pk>/submit-application/`
2. View `submit_application` проверял авторизацию и при её отсутствии возвращал `redirect("/auth/?next=...")` — то есть HTTP 302
3. `fetch()` по умолчанию **следует за редиректами автоматически** и в итоге получал HTML страницы входа с кодом 200
4. JS-обработчик не проверял статус ответа и вставлял полученный HTML прямо в `.apply-area` карточки через `innerHTML`

Существующая защита через `HX-Redirect` работала только для HTMX-запросов (проверка `request.headers.get("HX-Request")`), но shared modal использует чистый `fetch()`, а не HTMX.

### Исправление — три уровня

**Сервер** (`src/web/apps/frontend/views/applications.py`):
Для `fetch()`-вызовов при отсутствии авторизации теперь возвращается:
```python
JsonResponse({"error": "unauthenticated", "redirect": "/auth/"}, status=401)
```
вместо HTML-редиректа. HTTP 401 `fetch()` не преследует автоматически.

**JavaScript** (`src/web/templates/frontend/project_list.html`):
Добавлена явная проверка кода ответа перед обработкой:
```js
if (res.status === 401) {
    return res.json().then(function(data) {
        window.location.href = data.redirect || '/auth/';
    });
}
if (!res.ok) { throw new Error('server_error'); }
```

**Глобальная защита** (`project_list` + `project_detail`):
BUG-001 и BUG-002 взаимосвязаны: после закрытия доступа к странице каталога (BUG-001) анонимный пользователь физически не может дойти до кнопки «Откликнуться». BUG-002 тем не менее исправлен независимо — на случай прямых API-запросов.

### Нужно ли повторное внимание
Наблюдения не требует. Однако при добавлении новых `fetch()`-вызовов к закрытым endpoint'ам нужно помнить: всегда проверять `res.ok` / `res.status` перед вставкой ответа в DOM.
