# 0. AUTH FLOW

### 1. Вход / регистрация

`auth.html`

- login / password
- регистрация
- обработка ошибок
### Роли пользователя

- STUDENT
- CUSTOMER
- MODERATOR

определяют доступные действия и элементы UI

# 1. STUDENT FLOW

## Цель:

найти проект → откликнуться → попасть в команду


## Экраны

### 1. Экран: Каталог проектов

`project_list.html`

#### UI:

- список проектов
- статус (badge)
- поиск / фильтр

#### Backend:

```http
GET /projects/
GET /search?q=...
```

#### Логика:

- отображаются только `PUBLISHED`
- если нет проектов → empty state

#### Переходы:

```
project_list → project_detail
```

### 2. Экран: Страница проекта

`project_detail.html`

#### UI:

- title
- description
- tags
- команда (accepted / team_size)
- статус

##### UI состояния:

- если уже откликнулся → "Вы уже откликнулись"
- если STAFFED → "Команда набрана"
- если не PUBLISHED → кнопки нет


#### Действия:

##### Откликнуться

```http
POST /applications
```
body:

```json
{
  "project": id,
  "motivation": "..."
}
```

##### После действия:

- toast "Заявка отправлена"
- обновление UI
#### Условия:

```
status = PUBLISHED
accepted_participants_count < team_size
пользователь ещё не откликался
```
#### Переходы:
```
project_detail → application_list
```


тут ключевой UX:
- описание
- теги
- команда
- статус проекта  (DRAFT / ON_MODERATION /  PUBLISHED / STAFFED / REJECTED)
- кнопка "Откликнуться"  (доступна только если статус PUBLISHED)
#### Дополнительно (внутри project_detail):

`apply_form (modal / inline)`

используется для:
- ввода мотивации
- отправки заявки

после отправки:
- отображается уведомление (toast)
- обновляется статус заявки
### 3. Экран: Мои заявки

`application_list.html`
#### Backend:

```http
GET /applications
```
#### UI:

- проект
- статус
- дата
#### Статусы:

- SUBMITTED
- ACCEPTED
- REJECTED
#### Empty state:

- "У вас нет заявок"

# 2. CUSTOMER FLOW

## Цель:

 создать проект → набрать команду
## Экраны:

### 1. Экран: Мои проекты

`project_list.html`

#### Backend:

```http
GET /projects?owner=me
```

#### UI:

- список проектов
- статус (badge)
- количество участников (accepted / team_size)

#### Действия:

##### Создать проект

```http
POST /projects
```

→ переход на `project_form.html`

#### Логика:

- отображаются только проекты пользователя
- сортировка по дате (новые сверху)
#### Empty state:

- "У вас пока нет проектов"
- кнопка "Создать проект"
#### Переходы:

```
project_list → project_detail
project_list → project_form (create)
```

### 2. Экран: Создание / редактирование проекта

`project_form.html`

#### Backend:

```http
POST /projects              → create
PUT /projects/{id}          → update
PATCH /projects/{id}        → partial update
```

#### UI:

Форма:

- title
- description
- tech_tags
- team_size

#### Действия:

##### Сохранить

- create → POST
- update → PUT/PATCH

#### Ограничения:

```
редактирование запрещено если статус:
- PUBLISHED
- STAFFED
```

#### После действия:

- toast "Проект сохранён"
- редирект → project_detail
#### Ошибки:

- валидация полей
- отображение ошибок под инпутами


### 3. Экран: Страница проекта (customer view)

`project_detail.html`


#### Backend:

```http
GET /projects/{id}
```
#### UI:

- title
- description
- tags
- статус
- команда (accepted / team_size)

#### Действия:

##### Редактировать

```
→ переход на project_form.html
```
##### Удалить

```http
DELETE /projects/{id}
```

##### UX:

- confirm modal
- после → редирект в project_list
- toast "Проект удалён"

##### Отправить на модерацию

```http
POST /projects/{id}/submit
```
#### Условия:

```
доступно только если status = DRAFT
```

#### После:

- status → ON_MODERATION
- toast "Отправлено на модерацию"
#### Ограничения:

если статус:
- PUBLISHED
- STAFFED

→ редактирование запрещено
→ кнопка submit скрыта
#### Переходы:

project_detail → project_form
project_detail → project_applications

### 4. Экран: Заявки на проект

`project_applications.html`

#### Backend:

```http
GET /projects/{id}/applications
```

#### UI:

Список заявок:
- студент
- мотивация
- статус
#### Действия:

##### Accept

```http
POST /applications/{id}/actions/review
```

```json
{
  "action": "accept"
}
```
##### Reject


```http
POST /applications/{id}/actions/review
```

```json
{
  "action": "reject"
}
```

##### Логика:

```
при ACCEPT:
- application.status = ACCEPTED
- project.accepted_participants_count++

если достигнут team_size:
- project.status = STAFFED
```

##### После:

- обновление статуса заявки
- увеличение accepted_participants_count
- если достигнут лимит → project.status = STAFFED

#### UX:

- disable кнопок во время запроса
- confirm при reject

#### Ограничения:

нельзя принимать:
если проект уже STAFFED
#### Empty state:

- "Нет заявок"

#### Pagination:

- поддерживается (page / size)
# 3. MODERATION FLOW

## Цель:

контролировать допуск проектов

## Экраны:
### 1. Экран: Модерация проектов

`moderation.html`

объединяет:

- список проектов (ON_MODERATION)
- просмотр проекта
- действия

внутри:

- список проектов (status = ON_MODERATION)
- выбор проекта
- просмотр деталей
- действия:

[ Approve ]
[ Reject + comment ]

после действия:

- изменение статуса проекта
- уведомление (toast)
#### Backend:

```http
GET /projects?status=ON_MODERATION
POST /projects/{id}/moderate
```

#### UI (единый экран):

Состоит из двух зон:

##### Список проектов (слева / сверху)

- title
- краткое описание
- дата создания
- автор

##### Просмотр проекта (справа / снизу)

- полный description
- tech_tags
- team_size
- информация об авторе

#### Поведение

клик по проекту → загружается detail в той же странице
#### Действия
##### Approve

```http
POST /projects/{id}/moderate
```


```json
{
  "action": "approve"
}
```

###### После:

status → PUBLISHED

- проект появляется в каталоге
- toast "Проект опубликован"
- удаляется из списка модерации


##### Reject

```http
POST /projects/{id}/moderate
```

```json
{
  "action": "reject",
  "comment": "Причина отклонения"
}
```

###### UI:

- обязательное поле comment
- confirm action

###### После:

status → REJECTED

- toast "Проект отклонён"
- проект исчезает из списка

---

#### Ограничения

только MODERATOR имеет доступ
только проекты со статусом ON_MODERATION

#### UX требования

- disable кнопок во время запроса
- loader при переключении проекта
- обработка ошибок (retry)

####  Empty state
"Нет проектов на модерации"

#### Обновление данных

после approve/reject:
- либо refetch списка
- либо удаление из UI

# СИСТЕМНЫЕ ТРЕБОВАНИЯ (UX / STATE)

## Защита от повторных действий

- disable кнопки во время запроса
- idempotency (на backend)
## Permissions

STUDENT:
- смотреть проекты (PUBLISHED)
- откликаться
- смотреть свои заявки

CUSTOMER:
- CRUD своих проектов
- отправка на модерацию
- просмотр заявок
- review заявок

MODERATOR:
- просмотр ON_MODERATION
- approve / reject
## Navigation

основные переходы:

- project_list → project_detail
- project_detail → application_list (после apply)
- project_list (owner=me) → project_detail
- project_detail → project_applications
- moderation → project_detail (внутри)

глобальная навигация:

- projects (project_list)
- my projects (owner=me)
- applications
- profile

## Статусы проектов

- DRAFT
- ON_MODERATION
- PUBLISHED
- STAFFED
- REJECTED

используются в:

- project_list
- project_detail
- moderation

### Ограничения редактирования

редактирование запрещено если статус:

- PUBLISHED
- STAFFED
- ARCHIVED

### Поведение статуса STAFFED

- проект отображается в каталоге
- отклик невозможен
- отображается:
→ "Команда набрана"
или disabled кнопка
### Поведение после PUBLISHED

варианты:

- либо редактирование запрещено полностью
- либо при редактировании → статус возвращается в DRAFT

## Статусы заявок

- SUBMITTED
- ACCEPTED
- REJECTED

используются в:

- application_list
- project_applications
## Ограничения действий

- отклик возможен только при статус = PUBLISHED, accepted_participants_count < team_size (Поведение при повторной заявке
  если пользователь уже откликнулся:  кнопка "Откликнуться" скрывается  или  отображается "Вы уже откликнулись")
- отправка на модерацию только из DRAFT
- редактирование запрещено если статус: PUBLISHED, STAFFED, ARCHIVED
## Empty states

должны быть обработаны для:

- нет проектов
- нет заявок
- нет результатов поиска
## Feedback

после каждого действия:

- apply
- create/update project
- accept/reject application
- moderation decision

→ показывается toast / уведомление

## Error states

должны быть обработаны:

- ошибка загрузки данных (project_list, project_detail)
- ошибка отправки формы (apply, create project)
- ошибка модерации / review

отображение:

- error message
- retry action

## Loading states

должны быть обработаны:

- загрузка списка проектов
- загрузка проекта
- отправка форм

отображение:

- spinner / skeleton
- блокировка кнопок на время запроса

## Pagination / infinite scroll

используется в:

- project_list
- project_applications

варианты:
- pagination (page, size)
или
- infinite scroll

## Confirm actions

требуют подтверждения:

- удаление проекта
- отклонение заявки
- отклонение проекта на модерации

реализация:

- confirm modal


## Обновление данных после действий

после:

- apply
- accept/reject
- moderation

используется:

- либо повторный запрос (refetch)
- либо оптимистичное обновление UI
