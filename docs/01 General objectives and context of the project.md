# 1. Общие цели и контекст проекта

Актуализировано: 2026-04-29.

## Цель

Система Digital Student Assistant предназначена для поддержки выбора и сопровождения студенческих проектов. Она покрывает путь от импорта проектных предложений и публикации каталога до подачи заявок, модерации, формирования рекомендаций и синхронизации данных с внешними сервисами обработки.

Фактическое состояние проекта фиксируется в исходном коде, `README.md`, `docs/api_scheme.md`, контрактах `docs/architecture/contracts/*.json`, тестах и эксплуатационных инструкциях.

## Проблема

Исходный процесс выбора проектов опирается на большие таблицы и внешние формы. Это создает несколько повторяющихся проблем:

- студентам трудно быстро найти релевантные темы среди сотен строк;
- данные переносятся вручную между таблицами, формами и административными процессами;
- статусы заявок и модерации не видны всем ролям в едином контуре;
- рекомендации и поиск трудно поддерживать без единой нормализованной модели проектов;
- внешние ML- и graph-компоненты не должны напрямую зависеть от внутренней базы данных основного веб-сервиса.

## Пользователи

- `student` - просматривает каталог, управляет интересами и избранным, получает рекомендации, подает заявки и отслеживает их статусы.
- `customer` - создает и редактирует проекты, отправляет их на модерацию, просматривает и рассматривает заявки на свои проекты.
- `cpprp` - модерирует проекты, управляет сроками и шаблонами, просматривает административные сводки и экспортирует данные.
- `staff` - выполняет технические и административные действия через Django admin и API, где такой доступ явно разрешен.

## Фактический контур выполнения

- `src/web` - веб-сервис Django + DRF и интерфейс на Django templates.
- `src/ml` - эталонный FastAPI ML-сервис для поиска и рекомендаций.
- `src/graph` - FastAPI graph projector с Neo4j.
- `apps.faculty` - зеркало данных преподавателей, публикаций и курсов, а также сопоставление проектов с научными руководителями.
- `infra` - Docker Compose, Nginx, сценарии развертывания, резервного копирования и восстановления.

## Текущий объем реализации

В текущем состоянии реализованы:

- вход в систему с подтверждением адреса электронной почты;
- ролевые личные кабинеты и API для `student`, `customer`, `cpprp`;
- каталог проектов, карточки, заявки, избранное, интересы;
- импорт EPP из XLSX через API и командную строку;
- сценарий обработки инициативных тем;
- фасад рекомендаций и поиска с резервным режимом при недоступности семантического ML-сервиса;
- outbox API доставки событий, контрольные точки, подтверждения обработки, повторное чтение и начальный снимок состояния;
- эталонные потребители событий для ML и graph-сервисов;
- зеркало данных преподавателей и сопоставление проектов с научными руководителями;
- Docker Compose для production/staging, инструкция по развертыванию, резервное копирование и материалы по наблюдаемости.
