# 5. Поддержка, развитие, эксплуатация

Актуализировано: 2026-04-29.

## Сопровождение

Поддержка проекта распределяется по зонам ответственности участников команды. Каждый участник сопровождает свой компонент, а при сбоях на границах сервисов подключаются владельцы смежных частей системы.

Для сопровождения используются:

- контрактные тесты API и событий;
- release gate из корня репозитория;
- эксплуатационные сценарии развертывания и резервного копирования;
- журналы приложений, health/readiness endpoints и метрики Prometheus.

## Возможное развитие

Планируемые направления развития:

1. аналитика для ЦППРП;
2. мобильный клиент или адаптированный мобильный интерфейс;
3. отказ от внешних форм в пользу собственного контура подачи и обработки данных;
4. production SSO `edu.hse.ru` после отдельного согласования интеграционного контракта;
5. вынос ML и graph-сервисов на отдельные хосты при росте нагрузки.

## Критерии успешной эксплуатации

Качественными критериями успеха считаются:

- выполнение нефункциональных требований из `03 NFR.md`;
- отсутствие критичных ошибок в основных сценариях выбора проекта и подачи заявки;
- воспроизводимое резервное копирование и восстановление PostgreSQL;
- положительная обратная связь от студентов, заказчиков проектов и сотрудников ЦППРП;
- сохранение совместимости с каноническими контрактами API и outbox-событий.

## Эксплуатационная защита данных

- Production VM должна иметь ежедневное ночное резервное копирование PostgreSQL через
  `scripts/install-backup-timer.sh prod`.
- Резервное копирование по умолчанию запускается в `03:20`, записывает gzip-dump в
  `/var/backups/dsa/prod/<timestamp>/postgres.sql.gz` и хранит 14 дней.
- Для ручной проверки используется `DSA_ENVIRONMENT=prod scripts/backup-stack.sh`.
- Восстановление выполняется через
  `DSA_ENVIRONMENT=prod scripts/restore-postgres.sh /var/backups/dsa/prod/<timestamp>/postgres.sql.gz`.
- Neo4j Community не включен в автоматическое онлайн-резервное копирование: архив активного volume
  может быть неконсистентным. Если graph-data станет критичной для production, потребуется
  отдельный maintenance-window dump/restore и регулярная проверка восстановления.
