# ADR-001: Выбор архитектурного стиля - модульный монолит и выделенный ML-сервис

**Статус:** принято
**Дата:** 2025-12-05
**Проект:** «Цифровой Ассистент Студента»

**Примечание по реализации (2026-04-28):** текущий репозиторий реализует это решение как модульный монолит на Django + DRF + Django templates в `src/web` с отдельными эталонными сервисами `src/ml` и `src/graph`. Обмен изменениями выполняется через outbox API доставки событий (`/api/v1/outbox/events/`, `/api/v1/outbox/events/ack/`, `/api/v1/outbox/consumers/{consumer}/checkpoint/`, `/api/v1/outbox/snapshot/`). Фактические типы событий зафиксированы в `docs/architecture/contracts/event_contract.json`, а правила работы потребителей - в `docs/architecture/contracts/outbox_delivery_contract.json`.

---

## Контекст

Проект «Цифровой Ассистент Студента» представляет собой веб-систему для поддержки выбора проектов КР/ВКР студентами ФКН с использованием:

- семантического поиска;
- рекомендательной системы по интересам;
- фильтрации по технологиям и направлениям.

Основные пользователи: студенты, внешние и внутренние заказчики, сотрудники ЦППРП.

Ключевые требования:

- **Функциональные (MVP):**
  - импорт и разбор ЭПП XLSX;
  - просмотр, фильтрация и поиск по проектам;
  - личный кабинет студента: избранное, интересы, рекомендации, заявки;
  - личный кабинет заказчика: создание и редактирование проектов, просмотр заявок;
  - личный кабинет ЦППРП: утверждение проектов, сроки, шаблоны.
- **Нефункциональные:**
  - пик нагрузки осенью: около 1.5-2 тысяч студентов и 1.5-1.7 тысяч проектов;
  - требуется надежное хранение данных о проектах и заявках;
  - для рекомендаций приемлема **eventual consistency**;
  - развертывание выполняется на серверах вуза, ограничения инфраструктуры пока неизвестны;
  - срок разработки составляет около 15 недель, команда включает 4 человека: ML/MLops, backend, data, frontend.

Необходимо выбрать архитектурный стиль, который:

- реалистичен для команды и сроков;
- поддерживает модульность по домену;
- позволяет развивать ML-часть независимо;
- не перегружает проект сложностью микросервисов.

## Решение

Принят архитектурный стиль:

> **Модульный монолит (Python) и выделенный ML/LLM-сервис для рекомендаций и семантического поиска.**

### Кратко

1. **Backend** - одно приложение на Python/Django + DRF, логически разбитое на доменные модули:
   - `users` - пользователи, роли, профили;
   - `projects` - карточки проектов, статусы, сроки, импорт из EPP XLSX;
   - `applications` - заявки, статусы и последовательность переходов;
   - `account` - ролевые кабинеты `student`, `customer`, `cpprp`;
   - `imports` / `outbox` - импорт данных и внешняя лента изменений;
   - `recs` - фасад над ML-сервисом для поиска и рекомендаций;
   - `faculty` - зеркало данных преподавателей, публикаций, курсов и сопоставление с научными руководителями;
   - `frontend` - пользовательский интерфейс на Django templates.
2. **Операционная база данных** - реляционная база под Django ORM:
   - в dev/test используется SQLite, целевой рабочий контур остается на PostgreSQL;
   - модульные границы задаются кодом, API и контрактами, а не отдельными схемами баз данных.
3. **ML-сервис** - отдельный сервис:
   - отвечает за вычисление embeddings, индексов для семантического поиска и рекомендательной логики;
   - не хранит источник истины, а получает актуальные изменения через контракт `web` и outbox-интеграцию.
4. **Обновление внешних моделей чтения** - через outbox-ленту:
   - backend публикует доменные события `project.changed`, `application.changed`, `user_profile.changed`, `deadline.changed`, `import.completed`, `recs.reindex_requested`, `faculty.*`, `project_faculty_match.changed`;
   - ML-сервис, graph projector и потребители, связанные с `faculty`, читают эти события и обновляют свои индексы и модели чтения;
   - обеспечивается **eventual consistency**, достаточная для предметной области.

## Обоснование

1. **Почему не выбраны микросервисы:**
   - команда и срок ограничены;
   - микросервисы добавляют заметные накладные расходы: инфраструктура, развертывание, трассировка, контрактная совместимость;
   - предметная область тесно связана: проекты, заявки, студенты и ЦППРП постоянно взаимодействуют.
2. **Почему выбран модульный монолит:**
   - один исполняемый контур упрощает эксплуатацию на серверах вуза;
   - доменная модульность сохраняет структуру системы и снижает связность;
   - проще реализовать MVP по этапам Min1-Min4, не тратя ресурсы на распределенную архитектуру.
3. **Почему ML-сервис выделен отдельно:**
   - ML-часть имеет самостоятельный жизненный цикл: эксперименты, обучение, ресурсоемкие модели;
   - может требовать специфических ресурсов, например GPU;
   - выделение в отдельный сервис упрощает работу ML/MLops-команды без изменения основного приложения.
4. **Почему используется outbox и внешние сервисы:**
   - модульность фиксируется через API-контракты и доменные события;
   - ML- и graph-сервисы можно развивать независимо, не дробя основной backend преждевременно.

## Последствия

### Положительные

- Быстрая и реалистичная разработка MVP при сохранении архитектурной дисциплины.
- Возможность развивать и масштабировать ML-часть независимо от остального кода.
- Меньше инфраструктурной сложности по сравнению с полноформатными микросервисами.
- Четкие доменные границы в коде и базе данных упрощают сопровождение и развитие системы.

### Отрицательные и рискованные стороны

- Сохраняется единый процесс: нельзя гибко масштабировать отдельные доменные модули без их выделения в самостоятельные сервисы.
- Требуется дисциплина команды: нельзя нарушать модульные границы и накапливать лишние перекрестные зависимости.
- Нужна дисциплина сопровождения outbox-семантики: идемпотентность потребителей, подтверждения обработки, контрольные точки и повторное чтение.

## Рассмотренные альтернативы

1. **Простой монолит:**
   - проще на старте, но при добавлении ролей, сценариев перехода и интеграций быстро превратится в трудно поддерживаемую систему;
   - хуже подготовлен к долгосрочному развитию;
   - при правильной организации модульного монолита в будущем можно будет перейти к сервисной декомпозиции, но на текущем этапе польза такого шага неочевидна.
2. **Микроядерная архитектура:**
   - была бы уместна, если бы платформа превращалась в центральный узел для большого числа дополнительных сервисов, но сейчас такой сценарий не просматривается;
   - на текущем этапе важнее показать работоспособный продукт, чем усложнять систему на стадии проектирования;
   - отдельные идеи микроядерного подхода при необходимости можно локально использовать в подсистеме рекомендаций.
3. **Полноценная микросервисная архитектура:**
   - избыточна для текущих масштабов и сроков;
   - требует сложной инфраструктуры и значительно увеличивает нагрузку на команду.
