# 2. Функциональные требования (что система должна делать)

1. **Какие функции точно должны быть в MVP?**
   1. Минимум 1:
      1. подтягиваем google sheet / upload xlsx
      2. парсим
      3. и предоставляем для фильтрации / семантического поиска

   2. Минимум 2:
      1. Минимум 1
      2. Клиент студента
         1. Вход
         2. Возможность отмечать понравившиеся проекты (избранное)
         3. Возможность отметить свои "Научные интересы" (шаблон / ?свободная форма?)
         4. Рекомендованные проекты (система рекомендации по интересам)
         5. Вся документация по курсовой доступная для скачивания (ссылка на документ с сайта цппрп + авто скачивание)

   3. Минимум 3:
      1. Минимум 2
      2. клиент минимального функционала для заказчка (внутреннего / внешнего)
         1. CRUD на создание карточки проекта (по сохранении изменений отправка на модерацию администраторам ЦППРП)
         2. просмотр заявок на проект от студентов
      3. клиент для администратора цппрп
         1. утверждение проекта для ОП
         2. доступ к заявкам утвержденных заказчиком
         3. Установка сроков и дедлайнов
      4. клиент студента
         1. добавлена возможность оставлять заявку на проект
         2. можно посмотреть статус своих заявок на проекты
      5. Экспорт данных

   4. Минимум 4:
      1. Минимум 3
      2. Возможность вместе с научным руководителем подать инициативную тему
      3. ...

2. **«плюшки» для будущих версий?**
   1. ТГ бот напоминалка / отслеживание статуса заявки
   2. проекция множества смыслов проектов на трехмерное пространство для визуализации областей и более наглядного выбора области интересов / можно посмотреть свою область интересов в этом пространстве и увидеть пересечение с существующиеми темами или предложить свою
   3. Автогерации всех форм и документов для подачи в ЦППРП
   4. ...

3. **Какие роли пользователей и права доступа нужны?** Например, студент, преподаватель, администратор, внешний заказчик — и что каждый из них может делать.

4. **Каковы бизнес‑процессы и workflow для ключевых сценариев?** Рассмотрим два трека:
   1. Проект от заказчика
      1. Создание $\rightarrow$ утверждение для ОП $\rightarrow$ появляется в области видимости студентов $\rightarrow$ подача заявок $\rightarrow$ просмотр заявок $\rightarrow$ утверждение / отклонение $\rightarrow$ передача данных об утверждении студента на проект в ЦППРП $\rightarrow$ если все места укомплектованы проект помечается флажком и попадает в фильтр "укомплектованные проекты".
   2. Инициативный проект
      1. Заполненеи шаблона заявки $\rightarrow$ указание научного руководителя ФИО (даже, если его нет в системе) и участников (даже, если его/их нет в системе) $\rightarrow$ отправка на согласование в ЦППРП $\rightarrow$ получение обратной связи $\rightarrow$:
         1. в случае успеха $\rightarrow$ добавление проекта в общую видимость студентам (если есть места, если нет, то флан "укомплектован")
         2. в случае неудачи $\rightarrow$ возможность ознаковиться с комментарием (комментарий должен быть строго более 100 символов, чтобы пояснение от ЦППРП было развернутым иначе нельзя они не должны иметь возможность поставить отказ) $\rightarrow$ доработка возвращенного заявления $\rightarrow$ повторная оправка и так по кругу
